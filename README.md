# System-Analysis-and-Design

# Briefly summarize the DriverPassproject. Who was the client? What type of system did they want you to design?

DriverPass was a place where people could learn how to drive. They wanted to provide a system that allowed students to review material and schedule sessions with a driver.

#What did you do particularly well?

I feel that I was able to translate the needs from the company into a decent chart. With the chart I created, I would hope the system turns out great.

#If you could choose one part of your work on these documents to revise, what would you pick?

After receiving feedback from my classmates, I would revise the early roles. In the System Design Document, I ended up combining the Admin role and Big Boss role. I would combine the two in the Business Requirements Document.

#How did you interpret the user's needs and implement them into your system desgin? Why is it so important to consider the user's needs when designing?

I did my best to put alot of the features requested into the system design. There was only one design that wouldn't function as having the system be available offline would have caused syncing issues. Since the user will be using the system the most, it is very important to consider how they will interact with the system and try to find any flaws with it.

#How do you approach designing software? What techniques/strategies would you use in the future to analyze and desgin a system?

While doing the project, I took notes on any features mentioned during the interview. After the interview, I sorted them out in the order they would appear in. For example, DriverPass wanted customers to be able to choose a driving package. In order to choose a package, they would first need to create an account. In the future, I would probably do the same thing since it seemed to work very well for me. I think I would like to have another person to brainstorm with to help define the system a bit better early on.
